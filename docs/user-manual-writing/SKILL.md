# User Manual Writing

## 核心定位

這個 skill 用來產出「可以直接給人使用」的前端手冊，而不是只有大綱或流水帳。
你要先知道這份手冊給誰看、要解決什麼任務、內容要做到多完整，再用真實頁面、
真實流程與真實截圖把內容寫到可交付。

這個 skill 的預設水位如下：

- 內容深度：完整版
- 結構方式：讀者角色 + 平台雙維模板
- 截圖規則：真實截圖 + 圖說 + 關鍵點標注規則
- 交付內容：正文 + 附錄 + 質量檢查清單
- 約束風格：平衡型，不死板，但不能空泛
- 規模控制：執行時先問使用者；未回答時預設中等規模

## 適用場景

- 一般使用者的產品使用手冊
- 新人 onboarding 或快速上手指南
- 維護者 / 值班人員的排查手冊
- 後台管理台操作手冊
- 桌面端或行動端 UI 流程說明
- 需要真實截圖、圖說與細節說明的功能教學
- 需要釐清邊界條件、常見錯誤與排查建議的完整文件

## 工作原則

1. 先確認讀者與任務，再決定內容組織。
2. 先打開真實介面並操作，再寫說明。
3. 先用 Markdown 把內容寫穩，再依需求轉成 docx、PDF 或 HTML。
4. 文字必須對齊真實 UI 文案、按鈕名稱、頁面路徑與狀態提示。
5. 不要捏造截圖、功能、欄位、錯誤訊息或畫面行為。
6. 如果需要追問，請一次只問一個聚焦問題，避免把使用者淹沒。
7. 如果資訊不完整，先做合理推斷，但要把假設寫出來。
8. 若最終要輸出 PDF，避免在反引號中放中文 UI 文案，也避免使用 emoji。
9. 若當前對話裡已有其他手冊類 skill 鎖定了同一題答案或已提出下一題，直接復用，
   不要把同一個確認行或同一題選項再說一次。

## 啟動時的確認順序

開始撰寫前，先從當前對話、專案檔案與可操作環境中提取資訊。只有在缺口會直接影響
手冊品質時才追問，而且一次只問一題。

優先確認以下內容：

1. 產品 / 頁面 / 平台是什麼
2. 目標讀者是誰
3. 手冊類型是使用手冊、管理手冊、維護手冊還是 onboarding
4. 交付格式是 Markdown、docx、PDF 還是 HTML
5. 是否需要真實截圖
6. 手冊規模是小規模、中等規模還是大規模

如果使用者沒有回答第 6 題，預設採用中等規模。

## 與相鄰手冊 skill 的去重規則

這個 skill 可能和其他手冊 / guide 類 skill 同時被載入。此時要避免重複發問：

1. 如果當前對話已明確鎖定讀者、規模、格式或截圖需求，直接沿用，不要重問。
2. 如果上一則 assistant 訊息已經提出下一題，除非使用者要求重貼選項，否則不要
   把同一題再重複一次。
3. 每一則澄清訊息最多只能包含一個待回答問題區塊。
4. 若另一個更專門的手冊 skill 已清楚主導當前任務，這個 skill 應配合延續，
   不要再自起一套平行的澄清流程。

## 規模定義

手冊規模不是硬性頁數檢查，而是內容體量與章節密度的指引：

- 小規模：10 頁以內，聚焦單一主流程與必要補充
- 中等規模：20 頁以內，覆蓋主流程、常用功能、例外情況、附錄與檢查清單
- 大規模：30 頁以內，適合複雜產品、多角色或多平台組合手冊

若使用者只說「你看著辦」，採用中等規模。

## 執行流程

### 1. 先鎖定手冊邊界

不要把整個產品一股腦塞進去。至少整理出：

- 目標任務：讀者完成什麼才算成功
- 入口位置：從哪裡開始
- 主要流程：核心步驟有哪些
- 補充流程：設定、權限、匯出、異常、排查、FAQ 等
- 本輪不寫什麼：哪些內容先排除，避免失焦

### 2. 先讀現有內容，再進真實介面

先看現有文件、路由、頁面檔、測試說明、README 與既有截圖，避免手冊與產品脫節。
只要可以操作真實產品，就必須實際打開頁面驗證，不要只靠程式碼猜流程。

優先順序：

1. 使用者提供的實際網址、測試帳號、啟動方式
2. 專案內現有文件、README、測試說明、路由與頁面檔
3. 可操作的本地開發環境、staging、模擬器或桌面 App
4. 若以上都沒有，再退回靜態素材或既有截圖

### 3. 走一次完整主流程

至少完成一輪核心任務，並記錄：

- 每一步看到的畫面
- 關鍵按鈕 / 欄位 / 狀態提示
- 前一步和後一步的差異
- 成功後的結果
- 容易卡住的點
- 需要補充提醒的地方

如果產品有角色差異，例如一般使用者、管理員、維護者，先確認本輪服務哪一種，
不要把不同角色的流程混在一條主線裡。

### 4. 擷取真實截圖並寫圖說

若需求包含截圖，圖片必須來自實際操作。每張圖都要有用途，不准只是堆圖。

截圖規則：

- 每張圖只服務一個步驟、一個區塊或一個狀態轉換
- 優先截關鍵操作前後，而不是整路每一步都截
- 圖片檔名使用可讀命名，例如 `login-page.png`、`chat-send-message.png`
- 頁面過長時拆成多張圖，不要硬塞一張看不清的全頁圖
- 如果某個操作的前後差異很重要，截兩張圖而不是用文字硬講
- 截圖後立即比對手冊步驟，確認圖與文字一致
- 若畫面含敏感資料，先避免使用真實私密資訊

圖說規則：

- 先說這張圖的用途，例如「登入後首頁，用來確認導航是否正常載入」
- 再說讀者要看哪裡，例如「請先看右上角的新對話按鈕」
- 如有狀態變化，再補一句前後差異，例如「這個畫面和送出訊息前相比，多了助手回覆」

若無法取得截圖，不要假裝有圖。請明確標註：

- 哪些畫面已驗證並附圖
- 哪些畫面未能取得
- 缺口原因是什麼

若最終格式是 HTML 或 PDF，還要額外確認：

- 中間 HTML 真的有 `<img>` 標籤，而不是只剩 `!圖片說明` 這類 Markdown 原文
- 圖片路徑在導出時可被正確解析

### 5. 按讀者角色 + 平台組織手冊

這個 skill 的默認結構不是單一模板，而是雙維度選擇：

- 讀者角色：一般使用者 / 管理員 / 維護者
- 平台類型：Web / 桌面端 / 行動端

先選讀者，再選平台，然後再決定章節順序。需要模板時，讀取：

- `references/manual-templates.md`
- `references/quality-checklists.md`

### 6. 撰寫正文

預設輸出 Markdown。只有在使用者明確要求 docx、PDF 或 HTML 時，才改成交付指定格式。
即使最後不是 Markdown，也先用清楚章節結構整理內容，再轉換輸出。

正文至少要包含以下內容：

- 文件目的與適用對象
- 手冊規模與覆蓋邊界
- 使用前準備
- 主要任務流程
- 重要畫面與功能說明
- 常見問題 / 失敗場景 / 排查建議
- 已驗證與未驗證範圍

若是完整版，還應補上：

- 角色差異或權限前提
- 邊界條件與風險提醒
- 例外流程或失敗場景
- 術語或關鍵名詞說明

若最後要輸出 PDF，正文撰寫時還要遵守：

- 中文 UI 名稱優先用粗體，不要放進反引號
- 反引號優先留給 ASCII 路徑、URL、檔名、API 名稱與變數名
- 盡量不用 emoji 或特殊圖示字元

### 7. 補附錄

除正文外，默認要有附錄。附錄至少擇要包含：

- 驗證環境
- 截圖清單
- 文件假設
- 未驗證範圍
- 相關路徑 / 頁面 / 入口索引

### 8. 補質量檢查清單

這個 skill 的交付不是「寫完就算」，而是要能被複查。請在文件末尾或附錄中加入簡短但
實用的質量檢查清單，例如：

- 目標讀者是否清楚
- 主要流程是否可跟著完成
- 圖片與文字是否對齊
- 是否交代已驗證與未驗證範圍
- 是否寫清楚常見錯誤與排查方向

## 不同輸出格式的處理方式

### Markdown

- 作為預設輸出格式
- 保留清楚標題層級、圖片相對路徑、步驟編號與附錄

### docx

- 先完成 Markdown 定稿
- 再輸出為適合分享或列印的 docx
- 保留標題層級、圖片位置、圖說與檢查清單

### PDF

- 先完成 Markdown 或 HTML 定稿
- 再轉成 PDF
- 注意截圖尺寸、分頁斷點與可讀性
- 不要假設任何 Markdown 轉換器都支援圖片語法
- 優先使用 `scripts/render_manual.py` 生成 HTML / PDF
- 生成前先讀 `references/export-guidelines.md`
- 若腳本警告有 emoji 或中文 code span，先回頭修正文稿再導出

### HTML

- 適合做可直接打開的互動式手冊或內部知識頁
- 保留章節導航、圖片、圖說、提醒與質量檢查區塊
- 若是從 Markdown 轉來，先確認圖片語法已正確變成 `<img>`

## 導出指引

若最終交付是 HTML 或 PDF，優先使用：

- `scripts/render_manual.py`

這個腳本會做三件事：

1. 解析 Markdown 圖片語法並轉成真正的 `<img>`
2. 把相對圖片路徑解析成可供 HTML / PDF 使用的實際檔案路徑
3. 對 PDF 導出前最常見的問題提出警告，例如：
   - emoji
   - 中文被放進反引號

建議命令：

```bash
python3 scripts/render_manual.py /path/to/manual.md \
  --html /path/to/manual.html \
  --pdf /path/to/manual.pdf
```

如果你要更嚴格地攔截風險，使用：

```bash
python3 scripts/render_manual.py /path/to/manual.md \
  --html /path/to/manual.html \
  --pdf /path/to/manual.pdf \
  --fail-on-warning
```

需要更細的規則時，讀取 `references/export-guidelines.md`。

## 交付前自查

交付前至少完成以下檢查：

- 手冊語言與使用者要求一致
- 格式與使用者指定一致
- 章節順序符合讀者的工作流
- 手冊規模符合使用者選擇，若未選則符合中等規模
- 截圖檔名、引用路徑與文件內容一致
- 圖說清楚說明用途、關注區域與前後變化
- 步驟與實際畫面、按鈕名稱、頁面名稱一致
- 沒有捏造未驗證的功能與畫面
- 已清楚標出假設、限制與尚未驗證範圍
- 已附上附錄與質量檢查清單
- 若輸出 HTML / PDF，已確認圖片不是以 `!圖片說明` 文字殘留
- 若輸出 PDF，已確認沒有中文 UI 名稱被 code font 變成方框問號

## 推薦輸出骨架

若使用者沒有指定特殊模板，可優先採用這個順序：

1. 文件目的與適用對象
2. 手冊規模與覆蓋範圍
3. 使用前準備
4. 快速開始 / 主要任務流程
5. 核心功能與畫面說明
6. 常見問題、失敗場景與排查建議
7. 附錄
8. 質量檢查清單

若是維護手冊，請把「環境確認」與「排查步驟」提前；若是一般使用者手冊，先讓讀者
完成第一個任務，再補充進階功能；若是行動端或桌面端，補上平台差異。

## 邊界與回退策略

- 如果頁面無法啟動或網址無法訪問，先說明阻塞，再回頭整理已有資料，不要假裝完成實測
- 如果只能看到部分功能，先交付已驗證章節，並明確列出未驗證區域
- 如果使用者只要極簡版本，可以縮短篇幅，但仍要保留關鍵操作結果、圖說與限制說明
- 如果使用者要求面向多種角色，優先拆章節或拆文件，不要硬混
- 如果使用者沒有回答關鍵問題，一次只補問一題；若仍未回答，再採合理默認值並標註假設

## 範例任務

**範例 1：**
輸入：為一個 SaaS 後台管理台寫值班人員手冊，要有登入、搜尋、匯出、常見異常與截圖。

輸出方向：先確認值班人員的主要任務，再詢問手冊規模；實際打開管理台完成一輪查詢與匯出，
補上異常徵兆、圖說、附錄與質量檢查清單。

**範例 2：**
輸入：幫我為桌面端聊天工具寫新手上手指南，輸出 Markdown，附實際畫面。

輸出方向：先確認新手要完成的第一個任務，再詢問規模；完成登入、建立會話、傳送訊息、
查看歷史等流程，寫成完整但可交付的手冊。

**範例 3：**
輸入：幫我整理一份行動端 App 的 HTML 使用說明頁，要面向第一線客服。

輸出方向：先確認客服最常被問的場景與規模，再把操作步驟、截圖圖說、常見問題、未驗證範圍
與質量檢查清單整理成可直接開啟的 HTML 文件。
